# Security Policy

## Reporting a Vulnerability

The Steady Links project takes security seriously. If you believe you have found a security vulnerability, please report it to us **privately** before opening a public issue.

### How to report

* **GitHub Advisories**: Use GitHub's private vulnerability reporting feature at
  <https://github.com/notuntoward/obsidian-steady-links/security/advisories/new>

Please include the following in your report:

* A description of the vulnerability
* Steps to reproduce the issue
* The plugin version and Obsidian version you are using

### What to expect

* An acknowledgment within **7 days** of your report
* A fix or mitigation plan within **30 days** for confirmed vulnerabilities
* Public disclosure only after you have been notified and a fix has been released
* If the vulnerability is accepted, you will be credited in the release notes unless you prefer to remain anonymous

### What not to report

The following are **not** considered security vulnerabilities:

* Bugs that do not have a security impact
* Issues already known and addressed in the latest release
* Missing dependency updates that Dependabot has already flagged

## Supported Versions

Only the latest released version of Steady Links receives security updates. If you are using an older version, please update before reporting.

## Disclosure Policy

We follow a coordinated disclosure model:

1. Reporter submits a private vulnerability report
2. Maintainer acknowledges within 7 days
3. Maintainer investigates and confirms the vulnerability (within 14 days)
4. Maintainer develops and tests a fix
5. Fix is released as a patch version
6. A public advisory is published after the fix is available
